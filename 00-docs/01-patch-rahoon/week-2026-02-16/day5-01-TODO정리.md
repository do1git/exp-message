## 📝 프로젝트 요약
- **한 줄 요약**: 기존 채팅 엔진을 기반으로 실무 수준의 확장성과 범용성을 갖춘 **B2B 임베드형 메시징 솔루션**으로의 진화 방향 설정.

---

## ❗ 문제(AS-IS)
- **목적성 부재**: 핵심 엔진(채팅 기능)은 개발되었으나, 이를 적용할 구체적인 서비스 시나리오 미비로 프론트엔드 설계 및 UI/UX 방향성 난항.
- **아키텍처 모호성**: 서비스 특성(B2C, B2B 등)이 정해지지 않아 백엔드 고도화(성능, 보안, 데이터 격리 등)의 우선순위 결정 불가.
- **활용 방안 미비**: 단순 채팅 구현을 넘어 실제 비즈니스 가치를 창출할 수 있는 '서비스적 매력' 부족.

---

## 🎯 목표(TO-BE)
- **서비스 정체성 확립**: '채널톡' 모델과 같은 **임베드형 고객 상담/메시징 시스템**을 최종 지향점으로 설정하여 개발 로드맵 구체화.
- **범용적 모듈화**: 어떤 웹 환경에서도 스크립트 한 줄로 연동 가능한 **SDK(Software Development Kit)** 형태의 위젯 구조 지향.
- **기술적 신뢰성 확보**: 다수의 고객사가 동시에 이용하더라도 데이터 격리와 실시간성이 보장되는 **'튼튼한' 멀티 테넌시(Multi-tenancy) 백엔드** 구축.

---

## 🏗️ 설계/선택(Key Decisions)
- **인터페이스 전략**: 전체 페이지 방식이 아닌, 호스트 웹사이트에 자연스럽게 녹아드는 **플로팅 위젯(Floating Widget)** UI 채택.
- **데이터 격리 설계**: 여러 서비스(고객사)가 하나의 서버를 공유하더라도 데이터가 섞이지 않도록 `Service_ID` 기반의 논리적 격리 로직 적용.
- **실시간 통신 최적화**: 서버 자원을 효율적으로 관리하기 위해 Namespace/Room 기반의 채널 관리 및 Redis Pub/Sub을 활용한 분산 환경 고려.

---

## 📈 결과(Impact)
- **개발 생산성 향상**: 명확한 제품 정의를 통해 프론트엔드 컴포넌트 설계 및 백엔드 API 명세화의 기준점 마련.
- **비즈니스 확장성**: 단순 채팅을 넘어 '고객 응대', '사내 협업 도구' 등 다양한 도메인으로 즉시 변주 가능한 기술적 토대 확보.
- **엔지니어링 역량 증명**: 단순 기능 구현을 넘어 **확장성, 보안, 가용성**을 고민하는 시스템 설계자로서의 경험 축적.

---

## 🚀 추후 디벨롭해볼 내용
- **임베드 최적화**: 외부 웹사이트 삽입 시 발생하는 CORS 이슈 해결 및 위젯 로딩 속도 최적화를 위한 JS 번들 경량화.
- **상태 관리 자동화**: 사용자의 실시간 접속 상태(Online/Offline) 동기화 및 부재 시 자동 응답/푸시 알림 로직 구현.
- **관리자 전용 대시보드**: 들어온 문의를 순차적으로 배정하고 상담 통계(응대 시간, 만족도 등)를 시각화하는 Admin 페이지 개발.
- **메시지 영속성 고도화**: 대량의 대화 로그를 효율적으로 저장하고 검색할 수 있는 인덱싱 및 아카이빙 전략 수립.
