## 프로젝트 요약

- **한 줄 요약**: 서비스가 풀고자 하는 문제를 명확히 정의하고, 그에 맞는 개발 방향·우선순위·협업 기반을 수립함
- **키워드**: `목표 재설정`, `방향성`, `우선순위`, `리셋`

## 문제(AS-IS)

해당 서비스를 통해 풀고자 하는 **문제가 명확하지 않아** 아래와 같은 문제가 발생함.

1. **개발 및 성능 최적화 방법의 모호함**: 무엇을 최적화해야 하는지, 어떤 기준으로 개발해야 하는지 불명확
2. **개발 작업의 우선순위 부재**: 무엇을 먼저 해야 하는지 판단하기 어려움
3. **개발 흥미 감소**: 목표가 없어 동기 부여가 약해지고, 조금씩 개발에 대한 흥미가 줄어듦
4. **협업자 확보 어려움**: 위 이유들로 인해 "이 서비스가 왜 필요한가"를 설명하기 어려워 개발 협업자를 구하기 힘듦

## 목표(TO-BE)

문제(AS-IS)에 대응하여 아래 목표를 설정함.

1. **서비스가 풀고자 하는 문제 정의**: "이 서비스가 왜 필요한가", "누구의 어떤 문제를 해결하는가"를 명확히 문서화
2. **개발·성능 최적화 기준 수립**: 정의된 문제에 맞는 개발 방향과 최적화 기준(측정 가능한 지표) 확정
3. **개발 작업 우선순위 수립**: 문제 해결에 기여하는 정도에 따라 작업 우선순위를 정하고, 무엇을 먼저 할지 결정
4. **개발 동기·흥미 유지**: 명확한 문제 정의와 우선순위로 "왜 이걸 하는가"를 스스로 설명할 수 있게 하여 동기 부여
5. **협업자 확보 기반 마련**: 문제 정의·가치 제안을 문서화하여 "이 서비스에 함께할 이유"를 협업 후보에게 전달 가능하게 함

## 설계/선택(Key decisions)

| 구분 | 결정 |
|------|------|
| 서비스 방향 | 채널톡 스타일 B2C 연결 창구 (피드백 창구 + 참여형 콘텐츠 창구) |
| 최적화 대상 | 참여형 콘텐츠 창구 집중 (트래픽 집중 구간) |
| 우선순위 | 1순위 핵심 기능 구현 → 2순위 대기업 취업에 도움 되는 하고 싶은 것 |
| 동기·흥미 유지 | 작은 마일스톤 + 정기 점검 + 과부하 방지 |
| 협업 기반 | README + 한 페이지 가치 제안 + 작업중 태그 PR |

## 결과(Impact)

- **방향성 명확화**: "이 서비스가 왜 필요한가"에 대한 답이 정해져 개발·최적화 기준 수립 가능
- **우선순위 확보**: 핵심 기능 → 하고 싶은 것 순으로 판단 기준 확보
- **동기·흥미 유지**: 작은 마일스톤·정기 점검으로 지속성 확보
- **협업 가능성**: README·CONTRIBUTING·PR로 "함께할 이유" 전달 기반 마련

## 목표별 결정 과정

### 1) 서비스가 풀고자 하는 문제 정의

**고려한 후보**

| 후보 | 내용 | 장점 | 단점 |
|------|------|------|------|
| 채널톡 스타일 | 이용자와의 연결 창구 (B2C: 서비스 제공자 ↔ 이용자) | 사이드 프로젝트에 위젯 형태로 삽입 가능, 고객 문의·상담 창구로 활용 |  |
| 카카오톡 스타일 | 유저 간 채팅 플랫폼 (C2C: 유저 ↔ 유저) | 1:1/그룹 채팅 등 일반적 메신저 경험 | 내 상황에서 활용처가 제한적 |

**선택**: 채널톡 스타일 — 이용자와의 연결 창구

**선택 이유**:
- 사이드 프로젝트에 삽입하려는 용도에 전자가 더 적합함
- B2C 연결 창구로 쓰면 고객 문의·피드백 수집 등 실사용 시나리오가 명확함

**결정 결과**: 이 서비스는 **채널톡처럼 이용자와의 연결 창구**를 사이드 프로젝트에 제공하는 것을 목표로 하며, 핵심 기능은 다음 두 가지임.

1. **피드백 창구**: 유저와 내가 실시간으로 소통할 수 있는 창구
2. **참여형 콘텐츠 창구**: 참여형 콘텐츠를 통해 유저의 유입과 활동을 더 높일 수 있는 창구

---

### 2) 개발·성능 최적화 기준 수립

**고려한 후보**

| 후보 | 측정 지표 | 적용 범위 | 장점 | 단점 |
|------|----------|----------|------|------|
| 핵심 기능 전반 | 레이턴시, 처리량, 에러율 등 | 피드백 창구, 참여형 콘텐츠 창구 등 전체 | 균형 잡힌 품질 | 리소스 분산, 트래픽 집중 구간 대응 부족 |
| 참여형 콘텐츠 창구 집중 | 동시 접속, 처리량, 에러율 등 | 참여형 콘텐츠 창구 | 트래픽이 몰릴 구간에 집중 투자 가능 | 다른 기능은 상대적 우선순위 하락 |

**선택**: 참여형 콘텐츠 창구 집중 최적화

**선택 이유**:
- 참여형 콘텐츠는 이벤트·기능 발동 시 트래픽이 몰릴 가능성이 높음
- 이 구간에 최적화 기준을 높여두면 유입·활동 증대 목표 달성에 직결됨

**결정 결과**: 기존에는 핵심 기능 전반의 성능을 지속 높이는 방향이었으나, 이제는 **트래픽이 몰릴 참여형 콘텐츠 창구**에 최적화 기준을 높여 적용함.

---

### 3) 개발 작업 우선순위 수립

**고려한 우선순위 기준**

| 순위 | 기준 | 설명 |
|------|------|------|
| 1순위 | 핵심 기능 구현 | 피드백 창구, 참여형 콘텐츠 창구 등 서비스 핵심 기능 달성 |
| 2순위 | 대기업 취업에 도움이 되는 하고 싶은 것 | 그다음으로, 개인적으로 하고 싶으면서 대기업 취업에 도움이 될 만한 작업 우선 |

**선택**: 1순위 핵심 기능 → 2순위 대기업 취업에 도움 되는 하고 싶은 것

**선택 이유**:
- 핵심 기능이 없으면 서비스 가치가 없음
- 그다음에는 하고 싶은 것 중에서 취업 포트폴리오·경험에 도움이 되는 것을 하면 동기와 실리 모두 확보

**결정 결과**: **1순위는 핵심 기능 구현**, **2순위는 대기업 취업에 도움이 될 만한 하고 싶은 것** 순으로 우선순위를 정함.

---

### 4) 개발 동기·흥미 유지

**제안 방안**

- 작은 마일스톤으로 달성감 확보
- 2순위 하고 싶은 것 확보 (이미 우선순위에 반영됨)
- 정기 점검으로 "왜 하는가"를 되새김
- 과부하 방지, 지속성 우선

**결정 결과**: 작은 마일스톤 + 정기 점검 + 과부하 방지를 채택하여 적용함.

---

### 5) 협업자 확보 기반 마련

**채택 방안**

| 방안 | 내용 |
|------|------|
| README | "이 서비스가 왜 필요한가", "핵심 기능 2가지"를 README 상단에 명시 |
| 한 페이지 가치 제안 | 문제 정의 + 해결 방향 + "함께할 이유" 요약 (블로그·SNS·모집 글에 활용) |
| 진행 상황 공개 | GitHub 이슈·마일스톤, 주간 요약 등으로 투명성 확보 |

**결정 결과**: README + 한 페이지 가치 제안 + 진행 상황 공개 3가지를 채택하여 협업 기반을 마련함. 진행 상황 공개는 **작업중 태그를 달아둔 PR을 올리는 방식**으로 함.
